## Test 2: API Rate Limiting (Task 4)

**PASS** ✅

**Procedure:**
1. GET /login → extract CSRF token
2. POST /login with wrong password × 5 (with valid CSRF token per attempt)
3. Verify DB records 5 `LOGIN_FAILURE` entries
4. POST /login attempt 6 → verify HTTP 429

**Results:**

| Attempt | HTTP Status |
|---------|-------------|
| 1 (wrong password) | 302 (redirect to /login?error) |
| 2 (wrong password) | 302 |
| 3 (wrong password) | 302 |
| 4 (wrong password) | 302 |
| 5 (wrong password) | 302 |
| 6 (wrong password) | **429 Too Many Requests** |

**DB state after 5 failures:**
```sql
SELECT username, attempt_type, timestamp FROM login_attempts;
-- 5 rows: username='admin', attempt_type='LOGIN_FAILURE'
```

**Attempt 6 response body:**
```
Too many attempts. Try again in 14 minutes.
```

**Notes:**
- DB-persistent tracking survives server restarts
- Per-username lockout threshold: 5 attempts within 15-minute window
- The existing `BruteForceProtectionService` (in-memory) also independently tracks attempts — both systems co-exist
- After Test 2, admin user's in-memory BFA state needed a container restart to clear for subsequent tests

## Known Issues / Observations

1. **BFA consolidation applied (2026-05-29)**: The original `BruteForceProtectionService` write side-effect in `UserPrincipal.isAccountNonLocked()` was removed. All brute-force protection is now handled exclusively by Task 4's `LoginAttemptService` (DB-persistent). `UserPrincipal.isAccountNonLocked()` returns `true`; lockout is enforced by `LoginAttemptFilter` before authentication is attempted. Admin unlock: `DELETE FROM login_attempts WHERE username=?` (targeted) or `DELETE FROM login_attempts WHERE ip_address=?` (credential stuffing). No container restart needed.

2. **Client-side SHA3-512**: The app hashes passwords client-side before submission. Automated tests must replicate this (SHA3-512 the password).

3. **CSRF required**: All POST requests require a `_csrf` token obtained from GET /login. Automated tests must implement CSRF token extraction.

4. **Container image caching**: `docker compose restart` does NOT pick up a rebuilt image — use `docker compose up -d --force-recreate` or `docker stop/rm` + `docker compose up -d`.

5. **Large binary files in git**: GitHub warned about files > 50MB (JARs, APKs, EXE). These are under the 100MB hard limit and don't block operation.
